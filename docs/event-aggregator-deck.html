Product target

Move from places to time-aware weekend options.

Today

  • Users browse durable places like parks, museums, food, wellness, and shopping.
  • The planner ranks spots by vibe, age band, distance, cost, and hours metadata.
  • Share links preserve selected stops for group voting.

With events

  • Users see dated family events that are actually happening this weekend.
  • The planner can combine a timed event with nearby fallback stops.
  • Share links carry event date, time, venue, and source URL so voters can evaluate plans.

Coverage strategy

Use an explicit source registry, not one giant scraper.

Tier 1

Official civic calendars

City events, library systems, county parks, recreation departments, museums, and public agencies. These become the most trusted source class.

Tier 2

Structured feeds

ICS, RSS, JSON feeds, JSON-LD `Event` markup, and sitemap pages. These are cheaper to maintain than page-specific scrapers.

Tier 3

Ticketing APIs

Ticketmaster can fill paid concerts, performances, and venues. Eventbrite should be source-specific because broad public search access is not a reliable foundation.

Pipeline

The automation has five jobs.

1

Discover

Read `data/event-sources.json`, fetch each source with source-specific rate limits, and store raw snapshots for debugging.

2

Extract

Parse ICS, RSS, JSON, JSON-LD, or HTML selectors into raw event candidates without mixing product rules into scrapers.

3

Normalize

Convert every candidate into one schema with timezone, geocoding, city, category, age bands, cost, and source URL.

4

Validate

Drop stale, duplicate, adult-only, locationless, and unsourced records. Keep reject reasons for operator review.

5

Publish

Write `public/data/events.json`, run tests, build the app, and commit only meaningful feed changes.

Event schema

Use actual dates, not only day-of-week labels.

The existing seed uses recurring `daysOfWeek` plus `timeWindow`. That works for a preview rail, but real planning needs start and end datetimes so the app can rank, expire, and share events correctly.

  • `sourceUrl` is required so parents can verify details.
  • `confidence` separates official feeds from inferred data.
  • `rejectReasons` should live in the build report, not the user feed.

Quality gates

Parents should see fewer events, not noisier events.

Gate Required Reject example Why it matters
Source URL Yes No original event page Users need a place to verify time, tickets, and changes.
Future datetime Yes Missing start time or past event The app is making weekend plans, not an archive.
Bay Area city Yes Venue outside coverage registry Coverage should be explainable and auditable.
Family suitability Strict 21+, bar crawl, nightclub, adult comedy The app is positioned around kids and family planning.
Dedupe Fuzzy Same event from venue and ticketing API Duplicate events damage ranking and planner trust.

Automation design

Run small, observable refreshes instead of scraping blindly.

Hourly

Refresh high-change feeds for the next 14 days: libraries, city events, parks, ticketing APIs.

Daily

Refresh lower-change calendars and rebuild the full 60-day event file.

Weekly

Audit source coverage, dead links, selector failures, and cities with no active sources.

Manual

Support `workflow_dispatch` for emergency refreshes before weekends or product demos. 

npm run ingest:events
npm run validate:events
npm run test
npm run build

Build report

The hidden operator view is as important as the feed.

Coverage

Cities covered, cities missing sources, source count by county, and records by source type.

Freshness

Last successful fetch, failures by source, stale feed warnings, and next event date per city.

Reject reasons

Counts for duplicates, missing dates, no source URL, adult-only language, out-of-area venues, and parser failures.

Proposed output: `public/data/event-build-report.json` for diagnostics and future admin UI.

Implementation plan

A practical build path.

Slice 1
  • Add `data/event-sources.json` with Bay Area coverage.
  • Add source adapters for ICS, RSS, JSON-LD, and static JSON.
  • Normalize into the new event schema.
Slice 2
  • Add validation, dedupe, adult-only filtering, and reports.
  • Generate `public/data/events.json` from real sources.
  • Extend CI to validate and build the feed.
Slice 3
  • Add Events browse UI with city, date, cost, and age filters.
  • Let users save events into plans.
  • Update planner scoring for timed events.
Slice 4
  • Update Worker poll payloads for events.
  • Add Ticketmaster as optional keyed source.
  • Add coverage dashboard from build report.
