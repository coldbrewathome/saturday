End-to-End Flow

The important boundary: public/data/events.json should be a future-facing planning feed, not a historical database.

Inputs
Source Registry data/event-sources.json
  • Official libraries, parks, museums, festivals
  • Parser type and source metadata
  • Disabled sources with reasons
Fallback Templates data/event-templates.json
  • Recurring family-friendly programs
  • Used only when live source extraction returns nothing
to
Build Pipeline
Fetch Sources scripts/ingest-events.mjs
  • Loops through configured sources
  • Fetches HTML, JSON, RSS, ICS, LibCal, BiblioCommons
  • Records per-source status in the build report
Normalize Events scripts/eventPipeline.mjs
  • Extracts dates, title, venue, city, category
  • Infers age bands, cost, and time window
  • Dedupes and filters adult-only/calendar-chrome rows
Planning Window Guard generated date through +45 days
  • Past dated rows are not kept in the live feed
  • Future suggestions stay focused on upcoming events
to
Outputs and App Use
Public Event Feed public/data/events.json
  • Top-level dataset object
  • events[] contains normalized event rows
  • Current span: 2026-05-09 to 2026-06-23
Operator Report public/data/event-build-report.json
  • Source status, counts, HTTP errors
  • Useful for deciding which sources need new parsers
Planner and UI src/App.tsx and src/planner.ts
  • Verified events become planner candidates
  • Past, unverified, and template events are blocked from plans
  • Preference chips alter ranking and AI candidate payloads

Past Event Policy

Current Live Feed On each ingest, events are filtered to the generated date through the configured 45-day planning window. A past event naturally drops out on the next successful ingest.
Recommendation Do not keep past events in public/data/events.json. That file should stay small, current, and safe for suggestions.
Optional Archive If we want analytics later, write past/expired rows to a separate file such as public/data/event-history.json or data/event-archive/YYYY-MM.json. The planner should never read that archive.

App-side safety still matters: isActualPlanningEvent() rejects events that are unverified, templates, outside the 45-day horizon, not on a weekend, or already before today. A separate archive would be for reporting only.
