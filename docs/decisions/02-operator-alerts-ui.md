# ADR 02: Operator-Alerts Triage UI — Surface and Auth

- **Status:** Accepted
- **Date:** 2026-05-24
- **Decider:** Doer agent (kning to confirm)

## Context

`scripts/event-ops-agent.mjs` and `scripts/source-repair-agent.mjs`
emit `public/data/{metro}/event-operator-alerts.json` for each of the
14 metros, flagging broken sources, last-known-good fallbacks, and
zero-extracted feeds. Today the operator workflow is "open the JSON
files by hand, scroll, eyeball severities." Atlanta alone has 9 active
alerts; multiply by 14 metros and signal gets lost — sources silently
degrade and event coverage slips before anyone notices.

We need a place to:

1. See every metro's alerts in one view.
2. Filter by severity and metro.
3. Snooze noisy sources for one ingest cycle so the list stays scannable.

The roadmap task asks: standalone `/ops/alerts` route in the existing
app, a separate `worker/` admin page, or a static CLI report.

## Options considered

### Static CLI report
- **Pro:** Zero new code surface — extend `event-ops-agent.mjs` to
  print a colored table; no UI work. No auth question (terminal access
  == auth).
- **Con:** No interactive filters, no snooze action (would require
  writing a flag + re-running), no shareable URL for "look at this
  alert." Snooze persistence still needs a JSON file plus a CLI
  subcommand. We lose the ability to glance at it on mobile while
  on-call. The roadmap also asks for filter controls and a URL-state
  triage table, which terminal output can't deliver well.
- **Verdict:** Reject. Useful as a complement (and the existing
  agent's stdout already covers the "did the run pass?" case), but
  insufficient as the primary triage surface.

### Separate `worker/` admin page
- **Pro:** Clean isolation from the public-facing app bundle —
  operator-only HTML never ships to parent visitors. Worker already
  has `NEWSLETTER_ADMIN_TOKEN` and KV; the same pattern (`Authorization:
  Bearer …`) extends naturally. Snooze state could live in KV instead
  of a tracked-in-git JSON.
- **Con:** The alerts data lives in `public/data/` and is generated by
  Node scripts, not the worker. The worker would either need to fetch
  the same JSON files over HTTP (extra round-trip, plus a cross-origin
  story) or we'd duplicate the alerts into KV from CI. Snooze state in
  KV means `scripts/event-ops-agent.mjs` (Node, no Workers runtime)
  would need to call the worker API to read snoozes — another network
  hop for a script that runs locally. New surface = new auth, new
  routing, new TSX-or-HTML decision inside `worker/`. Heavier than the
  job warrants.
- **Verdict:** Reject for v1. Revisit if/when the alerts data itself
  moves into KV (e.g. for live ingest from production).

### Standalone `/ops/alerts` route in the existing app (chosen)
- **Pro:** Alerts JSON already ships as static assets under
  `public/data/{metro}/event-operator-alerts.json` (same origin as the
  app); a client-side route can `fetch()` all 14 in parallel with zero
  new server work. Reuses the existing hash-router pattern
  (`src/App.tsx` `readAppRoute`), build pipeline, and styles. Snooze
  state can live as a tracked JSON (`data/alert-snoozes.json`) that
  the existing Node scripts already know how to read — no new runtime
  bridge. URL-querystring filter state is trivial in React.
- **Con:** Operator UI is technically reachable by anyone who knows the
  hash. The data itself is already public (the JSON files are world-
  readable on famhop.com), so the "leak" is the UI affordance, not the
  data. Snoozing is a write — that one mutation does need an auth
  story (see below).
- **Verdict:** Accept.

## Decision

**Add `/ops/alerts` as a new hash route in the existing React app**
(`src/App.tsx`'s `readAppRoute`/`buildAppHash` switch). All read paths
(load alerts, filter, summary) are client-side and unauthenticated —
the underlying JSON is already public.

### Auth model

- **Read = unauthenticated.** The alert JSON files are already served
  publicly under `/data/{metro}/event-operator-alerts.json`. Gating the
  UI without gating the data is security theater. We accept that a
  curious visitor could land on `#/ops/alerts` and see source-health
  metadata; none of it is PII or a secret.
- **Write (snooze) = local-only for v1.** The snooze action does not
  hit the worker. It writes to `data/alert-snoozes.json` via a
  developer-facing path — concretely, the UI surfaces a "copy snooze
  payload" affordance (a JSON snippet the operator pastes into the
  tracked file) **or** a small Node helper invoked from CLI:
  `node scripts/snooze-alert.mjs <sourceId> --until=<ISO>`. The UI
  itself does not mutate any file. This sidesteps the auth question
  entirely for v1 while still delivering the snooze workflow on a
  weekly cadence (operator is at a terminal anyway when triaging).
- **Discoverability:** No nav link from the public app. The route is
  unlisted; operators know the URL. Add a short note in the
  `event-ops-agent.md` runbook pointing at it.

### Why not gate reads behind a token?

Two reasons:

1. The data is public. The agent already publishes
   `event-operator-alerts.json` to the same CDN that serves spots and
   events. Any token check on the UI route is bypassed by `curl
   https://famhop.com/data/atlanta/event-operator-alerts.json`.
2. Adding a token check to a client-only React route is meaningless
   (the secret would ship in the bundle). Real gating would require
   moving the data behind the worker — out of scope for v1 and
   contradicted by the existing publish pipeline.

If the alerts ever start carrying anything sensitive (subscriber
emails in a bounce alert, e.g.), revisit by moving both the data and
the UI behind the worker with the existing admin-token pattern.

### Snooze persistence (preview of the next task)

- File: `data/alert-snoozes.json` (tracked in git so snoozes survive
  across operators / CI).
- Schema (v1):
  ```json
  {
    "schemaVersion": 1,
    "snoozes": [
      { "sourceId": "terminal-west-atl", "until": "2026-05-31T00:00:00Z", "note": "venue site down — check next week" }
    ]
  }
  ```
- `scripts/event-ops-agent.mjs` reads this file when emitting alerts
  and either suppresses the alert or tags it `snoozedUntil` so the UI
  can grey it out. Expired snoozes (`until < now`) are ignored on
  read; a follow-up housekeeping pass can prune them.

## Consequences

- New hash route `#/ops/alerts` lives next to `#/browse` and
  `#/plans`. Bundle size grows by the size of the loader + table
  component (target: <8 KB gzipped).
- 14 parallel `fetch()` calls on route mount — acceptable; the same
  app already does multi-fetch on first paint for spots/events/plans.
- `data/alert-snoozes.json` becomes a tracked file. Routine snooze
  churn will show up in `git status` — acceptable given the weekly
  cadence and small file.
- No worker changes. No new secrets. No new deployment target.

## Rollback

If the route turns out to be the wrong surface (e.g. operator wants
push notifications instead of a pull-based dashboard), the route is
self-contained under a new directory (`src/ops/`) and one
`AppRoute["view"]` enum value. Delete the directory, remove the enum
value, drop `data/alert-snoozes.json` from git, and the public app is
unchanged.
